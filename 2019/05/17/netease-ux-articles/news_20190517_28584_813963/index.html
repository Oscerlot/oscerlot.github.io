Mobile Game Social System - Detailed Explanation of Friends Interface Interaction Design
17 May, 2019

Machine Translated from Chinese to English
Original article written by ThunderFire UX (NetEase)

The social system has always been an indispensable part of many games. Although it is not directly related to the core gameplay, it is a key system to increase player immersion and improve retention. Especially for MMO products with strong social interaction, the social system plays a key role. For example, our A Chinese Ghost Story mobile game is known for its excellent social design.

From the perspective of planning, when designing a social system, we usually consider how to meet the needs of players, how to promote the establishment of a player network, how to use the elements in the game to create a social space for players, and so on. As interaction designers, when we are involved in social system design, we need to consider more about how to improve system usability, reduce interruptions, and satisfy the rational layout of all functions.

Here, we focus on the main operating system in the mobile game social system - the friends interface, and sort out the interaction design elements of the friends interface.

I. Functional Block Breakdown

Divided by functional blocks, friends can be divided into these three parts:

  • Existing Friends
  • Find Friends
  • Message Notifications

The existing mobile game friends interfaces basically include these three parts. Depending on whether a certain game focuses on social interaction, the specific functions within each part will be different. Games that focus on social interaction often have more expanded functions, such as moments, message boards, social relationship recommendations, etc.

Block Description
Existing Friends Displays a list of current friends, allowing players to interact with them (e.g., chat, gift-giving, team up).
Find Friends Provides methods to search for and add new friends (e.g., by username, nearby players, recommended players).
Message Notifications Displays friend requests, system messages related to social interactions, and potentially mail (though mail is sometimes a separate system entirely).

The diagram above (not included here, as it requires a visual representation) shows the specific functions usually included in the friends block. It can be seen that there are quite a lot of function items included. Therefore, sorting and merging these function items is a prerequisite for rationally arranging the layout of the friends interface and making the overall hierarchy simple and clear.

1. First-Level Tab Division

Usually, the three major functional blocks are used as the three first-level tabs of the friends section. If there are more expanded functions, then second-level tabs are added. The message notification part is special. Mail notification is a relatively independent part. Some games will put it together with the friends block as an independent tab, and some games will separate it. This can be determined according to the specific planning direction of the game, and it has little impact on the friends block itself.

 

2. Friend Request List Entry

Although friend requests are message notifications, they are usually placed on the friends list page. This is because after a friend request is accepted, the friend will enter the friends list, which is logically related in terms of operation. Therefore, it is easier for players to understand that the request information is placed on the friends list page. And the importance of friend requests is relatively low, only requiring a prompt when there is request information. Usually, a button can be placed as an entry or used as a second-level tab.

 

3. Find Friends Functional Block Merging

In the “Find Friends” section, it can be seen that the search results, recommended friends, nearby people, and recent contacts all contain player lists and add friend operations, only differing in the specific information of the players.

 

Therefore, we can usually combine two of the blocks for display. For example, the recommended friends list is displayed in the default state, and after a search, this area displays the search results list.

Using this method of switching display content in the same area can more effectively utilize interface space and simplify the interface hierarchy. If there are multiple functional blocks at the same time, tabs can be added to clarify the logic.

 

II. Prompt Detail Design

In order to make the player’s operation smooth, in interaction design, it is necessary to exhaustively list all possible operation situations that the player may encounter, otherwise some details may be overlooked. There are also some design details that need attention in the friends section.

1. Nickname Display

The nickname is the beginning of the player’s establishment of social relationships and identification of identity in the game. Therefore, the player’s nickname needs to be clearly displayed first in the friends list. Due to limited space in mobile games, the length of the nickname is usually limited. However, when designing the layout of list information, we still need to consider long nicknames and multiple languages. Simple icons or levels can be placed before the nickname, but free space needs to be reserved after the nickname. Usually, the nickname is on a separate line, and other information is on a new line. In this way, when the nicknames of various players are of different lengths, it will not affect the display of subsequent information and keep the layout clean.

 

2. Prompt Information

Clear prompt information is the guarantee of smooth use by players, and there are also a series of prompt information that needs attention in the friends section.

For example, for nearby people, when the player turns off the positioning function, the list of nearby people is blank, and it needs to be prompted that the current positioning function is not turned on. In this case, we usually use positive guidance prompts such as “Turn on the positioning function to discover more players” with corresponding icons to guide players to use this function.

For another example, when the request list and search results are empty, it is necessary to pay attention to displaying corresponding prompt information such as “No friend requests” and “No corresponding friends found”. Without these prompts, a blank interface will make players wonder if it is stuck or if there is no content.

 

Clear interface hierarchy, functional block division, and detailed design details are aspects that we must consider when doing interaction design. This is the basic guarantee for satisfying usability and the premise for producing more innovative designs.
