Introduction to Data Desensitization Technology

Original article written by ThunderFire UX (NetEase)

With the advent of the big data era, data has become an important factor of production. For data controllers and data processors, how to maximize the value of data flow is the original intention and significance of data mining. However, the exposure of a series of information leakage incidents has made data security more and more widely concerned.

Data security plays an important role in the construction of a data warehouse. The leakage of private or sensitive data will cause serious damage to the property, reputation, personal safety, and legitimate interests of the data subject. Therefore, we need to strictly control access to the data in the warehouse. This requires classifying the sensitivity of the data itself into security levels. With the classification of data security levels, we can better manage data access control to protect data security.

But in the actual production process, the application scenarios will be more complex, and relying on access control alone, will be insufficient and other methods will be needed. Data desensitization is an effective method that can not only meet the needs of daily production, but also protect data security.

1. What is Data Desensitization?

Data desensitization refers to the transformation of certain sensitive information through desensitization rules to achieve reliable protection of sensitive and private data. In this way, the security level of the data itself can be downgraded, and the desensitized real data set can be safely used in development, testing, and other non-production environments, as well as in outsourcing or cloud computing environments.

But it should be noted that with the help of data desensitization technology, while erasing sensitive content in the data, it is also necessary to maintain the original data characteristics, business rules, and data correlation, to ensure that we will not be affected by desensitization in development, testing, and data analysis businesses, and to ensure the consistency and validity of data before and after desensitization.

2. Data Desensitization Strategies

Strategically, data desensitization is divided into Static Data Masking (SDM) and Dynamic Data Masking (DDM).

Static Data Desensitization

Static data desensitization is the “moving and simulated replacement” of data. The data is extracted for desensitization processing, and then distributed to downstream links for free access and read-write. The desensitized data is isolated from the production environment, meeting business needs and ensuring the security of the production database.

Static data desensitization is suitable for extracting data from the production environment, desensitizing it, and distributing it to scenarios such as testing, development, training, and data analysis. Sometimes we may need to copy the data from the production environment to the test and development libraries to troubleshoot problems or perform data analysis, but for security reasons, sensitive data cannot be stored in a non-production environment. In this case, sensitive data must be desensitized from the production environment before being used in a non-production environment. In this way, the desensitized data is isolated from the production environment, meeting business needs while ensuring the security of production data.

Dynamic Data Desensitization

Dynamic data desensitization performs desensitization processing in real time while accessing sensitive data. It can execute different desensitization schemes for different roles, different permissions, and different data types, thereby ensuring that the returned data is usable and secure.

Dynamic data desensitization is generally used in the production environment, because sometimes in different situations, different levels of desensitization processing are required for reading the same sensitive data. For example, different desensitization schemes may be executed for different roles and different permissions.

Differences in Technical Routes of Static Desensitization and Dynamic Desensitization

Static desensitization directly uses various desensitization algorithms such as masking, deformation, replacement, randomization, format-preserving encryption (FPE), and strong encryption algorithms (such as AES) to perform data masking disruption for different data types, and the desensitized data can be loaded into different environments according to user needs. Static desensitization can provide different loading methods such as file-to-file, file-to-database, database-to-database, and database-to-file. The exported data is stored in an external storage medium in a desensitized form, which actually changes the stored data content.

Dynamic desensitization accurately parses SQL statements to match desensitization conditions, such as: access IP, MAC, database user, client tool, operating system user, host name, time, number of affected rows, etc. After successful matching, it rewrites the query SQL or intercepts and protects the returned desensitized data to the application side, thereby achieving desensitization of sensitive data. In fact, the data stored in the production database has not changed.

3. Data Desensitization Solutions

The data desensitization system can define and write desensitization rules according to different business scenarios, and can perform data desensitization without landing for a certain sensitive field of a library table.

There are many ways to desensitize data, but no matter which method is used, it must be based on different application scenarios and follow the following two principles:

  • remain meaningful for application logic
  • sufficiently treated to avoid reverse engineer

Invalidation

The invalidation scheme processes the data to be desensitized by truncating, encrypting, hiding, etc. the data value of the field to desensitize the sensitive data and make it no longer valuable. Generally, special characters (*, etc.) are used to replace the true value. This method of hiding sensitive data is simple, but the disadvantage is that the user cannot know the format of the original data. If the user wants to obtain complete information, the user must be authorized to query. For example, if we replace the ID number with * to replace the real number, it becomes “220724 ****** 3523”, which is very simple.

Random Value

Random value replacement replaces letters with random letters, numbers with random numbers, and text with random text to change sensitive data. The advantage of this scheme is that it can retain the format of the original data to a certain extent, and this method is often imperceptible to users.

Data Replacement

Data replacement is similar to the previous invalidation method, except that special characters are not used for masking, but a set virtual value is used to replace the true value. For example, we set the mobile phone number to “13651300000”.

Symmetric Encryption

Symmetric encryption is a special reversible desensitization method. Sensitive data is encrypted through an encryption key and algorithm. The ciphertext format is consistent with the original data in logical rules, and the original data can be restored by decrypting the key. It is important to note the security of the key.

Average

The average scheme is often used in statistical scenarios. For numerical data, we first calculate their average value, and then make the desensitized value randomly distributed around the average value, thereby keeping the sum of the data unchanged.

Offset and Rounding

This method changes numerical data by random shifting. Offset and rounding ensure the approximate authenticity of the range while maintaining data security. It is closer to real data than the previous schemes and is of great significance in big data analysis scenarios. For example, the date field 2020-12-08 15:12:25 becomes 2018-01-02 15:00:00.

Data desensitization rules are often used in combination with multiple schemes in practical applications to achieve a higher level of security.

4. Conclusion

Whether it is static desensitization or dynamic desensitization, the ultimate goal is to prevent the abuse of private data within the organization and prevent private data from flowing out of the organization without desensitization. In the design and implementation of data desensitization schemes, it is more important to start from specific application scenarios for overall design, taking into account the important consideration dimension of data warehouse construction. The implementation of data desensitization can provide strong support for the promotion of organizational data security.
