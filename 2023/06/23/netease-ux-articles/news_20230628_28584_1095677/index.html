Fault-Tolerant Design in Games How to Apply Postel's Law
23 Jun, 2023

Machine Translated from Chinese to English
Original article written by ThunderFire UX (NetEase)

 

“Just like the Roman god Janus (and many politicians), every Web application has two faces: its ‘human face’ interacts with people, and the machine face interacts with computer systems.” —— “Your Website has Two Faces” by Lyle Mullican

Human thinking is different from that of machines. Human understanding of information is fuzzy, variable, and broad, while machines can only recognize data that conforms to a specific structure. How can the two understand each other and achieve a balance between input and output? This requires certain fault-tolerant design, so that the product can use “two faces” to interact with people and machines, respectively. This article will introduce the theoretical basis behind this - Postel’s Law, and introduce its application in game fault-tolerant design with examples.

1. What is Postel’s Law?

First, let’s understand the content and origin of Postel’s Law. Postel’s Law (Robustness Principle) is also known as the robustness principle. In 1980, computer scientist Jonathan Bruce Postel released an early specification of the TCP protocol. In the specification, he proposed this law:

Be conservative in what you send, be liberal in what you accept.

“Be conservative in what you send” means that the product should abide by certain specifications when displaying content to users; “be liberal in what you accept” means that when receiving user input, non-standard input should be allowed. The core idea it expresses is: the product should have a certain fault tolerance. This has become a design guide in the computer field, and has been widely used in game interaction/user experience design.

2. Allow Diversity, Tolerate Errors and Exceptions

When designing a game system, we cannot only allow players to enter the format that the machine can recognize, but also consider the diversity of users, accept and adapt to more flexible and diverse input and operations, and design corresponding processing methods.

Allow Players to Perform Arbitrary Operations

We may have had this experience. Sometimes, when we enter incorrect text or ambiguous content in the search box, the software can still give the desired search results, as if it has “figured out” our thoughts. In this process, the program does not limit input when facing the user. Even if the content is wrong or invalid, it will not crash or report an error. Instead, it will process the input content autonomously after receiving the input, showing a high degree of tolerance. We don’t need to delve into the program implementation behind this, we just need to know that this is in line with Postel’s Law.

The nickname box in Elden Ring also handles the input of spaces in a similar way. Players can enter spaces at will without any error prompts. But when the confirm button is pressed, the program will automatically process the content, delete the spaces at the beginning and end, and the extra spaces in the middle, so that the nickname becomes a format that conforms to the rules.

 

《Elden Ring》 Character Nickname Input Self-Processing

Input Method Adaptation

Games have a variety of input methods. Taking computer games as an example, the input methods can be a controller or keyboard and mouse. Many games can recognize these different input methods and give corresponding responses, so that they can tolerate and meet the different operating needs of players.

For example, Naraka: Bladepoint has adaptively processed the input method: when no controller is connected, it will default that the player is using keyboard and mouse operations, and the interface will also display keyboard and mouse prompts; when the player connects a controller, the game will automatically recognize and switch the operation mode to controller operation, and the interface prompts will also become controller icons accordingly.

 

《Naraka: Bladepoint》 Input Method Adaptation

Output Method Adaptation

Game products are usually not limited to running on one platform or one system. We need to adapt to screen resolutions, device types, and operating systems to reduce the impact of different device conditions on interface effects and operating experience. Taking Revelation M as an example, in addition to running on various mobile phones, it can also be played through an emulator. When we drag the interface to change to any size, we can see that the interface layout and component positions are automatically adjusted, and the interface can always maintain a reasonable and clear presentation effect.

 

《Revelation M》 Interface Adaptation

Exception Handling

Various abnormal situations such as network delay and disconnection may occur during the game running process, which is difficult to avoid. Therefore, it is also necessary to consider in advance and design the processing method when developing the game, which can prevent problems such as process loss and data damage caused by abnormal situations. Disconnect protection and reconnection, automatic saving and other functions are common exception handling methods:

If the controller is disconnected during the game process of Dead Cells, a “Controller Disconnected” prompt will pop up and the game will be automatically paused to avoid the character being out of control and affecting the game process. If the controller is reconnected, the prompt will disappear and the game will automatically resume.

 

《Dead Cells》 Controller Disconnection and Reconnection Handling

In Hogwarts Legacy, players can manually save the game, and the system also automatically saves frequently (there will be an icon prompt on the interface when saving). If you encounter a special situation during the game and don’t have time to save manually, don’t worry, you can find these automatic saves in the loading game interface, which largely avoids the loss of game progress/archive damage.

 

《Hogwarts Legacy》 Auto Save

3. Define Actionable Boundaries

Although we can use the game’s own inclusive design to accept various behaviors of players as much as possible, sometimes we still don’t want various strange operations to occur. This requires setting some boundaries to properly guide players and reduce the possibility of incorrect input.

Clear Definition Instructions

We can tell players the boundaries of operation by directly writing down the rules, so that players can autonomously filter out some wrong input situations. This method is more common in the interface for entering nicknames. The rules for nicknames are often more complicated, involving character type restrictions, character length restrictions, space processing, etc., and various input situations may occur. Different games also have different requirements for nicknames. Appropriately marking some rules will make it more convenient for players to enter content that meets the requirements.

Limited Options

We can limit the operation by reducing the freedom of input components and filtering out some options outside the boundary. For example, in the trading house of Revelation M, the value of “Purchase Quantity” requires players to enter pure numbers. When clicking on a number to enter, the system will force the numeric keyboard to pop up, avoiding the situation where players enter other characters.

 

《Revelation M》 Numeric Keyboard

Condition Constraints

Condition constraints are a commonly used design method in Metroidvania-like games. There are often some places on the map that cannot be reached in the early stage, and you need to obtain a certain ability or some props to explore. Through the gradual unlocking of various conditions, players can be guided to play according to the design expectations, enjoy the fun of constantly discovering new content, and also make the growth of the character and the rhythm of the game reach a relatively stable state.

At the exit of the novice map of Hollow Knight, this design is made. A bug is arranged to block the way. Players need to learn spell skills to pass, prompting them to return to explore other paths and improve their character abilities. It also prevents players from accidentally breaking into the late-game map in the early stage.

 

《Hollow Knight》 Using Character Abilities to Restrict Map Exploration

4. Operation Feedback and Correction

Defining boundaries can only prompt players what kind of operation is appropriate to a certain extent, but players can still perform some operations outside the boundary. Therefore, when players take incorrect or risky operations, they can also give some feedback or remedial measures to help players correct their operations autonomously.

Information Prompts

Information prompts are a commonly used feedback method. By giving prompts and guidance after players perform interactive operations, players’ operations are further corrected. The timeliness of information feedback is very important, and it needs to be pointed out immediately when an operation occurs. Hades has strict requirements on the steps of operation. For example, you must collect rewards before entering the next level. If you press the button to interact with the door without completing it, the interface will display “Collect Rewards!” information feedback, and at the same time, there will be a light spot sliding to the reward item to prompt and guide the player.

 

《Hades》 Operation Feedback and Prompts

Secondary Confirmation

Secondary confirmation is very common in games. It generally appears after some important operation steps, prompting players to the possible results of this operation, so that players can make a more cautious judgment of their operations, and prevent possible misoperation situations. For example, when clicking exit during the Devil May Cry 5 level, a secondary confirmation interface will pop up to ask, telling the player that if they exit at this time, they will continue from the last checkpoint when they restart, so that the player can confirm whether to return to the main menu.

 

《Devil May Cry 5》 Secondary Confirmation Interface When Exiting the Level

Revocable Operations

Although there is no “regret medicine” in the real world, the game world is imaginative, and anything can happen. Therefore, some games are very considerate to provide players with revocable operations, so that players can have the opportunity to make up for their mistakes after discovering them. For example, the store in Revelation Online has a repurchase function. After an item is sold, it will not disappear immediately, but will enter the repurchase list. If you accidentally sell the wrong thing, you can buy it back.

 

《Revelation Online》 Store’s Repurchase Function

Dark Souls 3 also has this design. If the player accidentally hits a friendly NPC and forms a hostile relationship, they can pray at the Statue of Velka in the game to restore the relationship with the NPC.

5. Summary

Postel’s Law tells us: the game needs to have two faces, one facing the player and the other facing the machine. The face facing the player needs to be open and inclusive, allowing the existence of diverse player operations and the input of various non-standard information; while the face facing the machine should follow the specifications and be cautious to avoid errors and confusion, so as to better convey the content to the player.

Fault-tolerant design in games is a very important part of the game experience. The application of Postel’s Law in games can help game designers design more stable systems and more user-friendly interfaces, improve the fault tolerance of game products, and thus further enhance the player’s game experience.
